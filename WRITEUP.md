# Stem Agent

## The problem and my approach

The challenge asked for a stem agent: a minimal agent that receives a class of problems, observes how they are solved, and builds itself into a specialized agent. The biological metaphor is precise. A stem cell does not start as a neuron, it becomes one by reading signals from its environment.

Most agents built today are born already specialized. The programmer writes the prompt, picks the tools, defines the flow. The agent works, but only works for what it was designed to do. What the challenge asked for was different: specialization that emerges from the process, not from the programmer.

My implementation solves this in three phases. First, a minimal agent observes the domain by researching how experts work, what tools they use, what distinguishes a good result from a bad one. Second, it synthesizes those observations into a concrete configuration: a specialized system prompt, tools, acceptance and rejection criteria. Third, it generates an executable agent that I never wrote. For a different domain (QA, Security, Code Review) the process would be identical and the result completely different.

---

## The architecture

The pipeline has three separate phases with explicit contracts between them.

**Observe** uses GPT-5 to research how experts work in the domain. The prompt is intentionally domain-agnostic because it does not mention Technical Research or any specific domain. The same file works for any domain without modification. The output is free text validated with a minimum length to guard against degenerate responses.

**Synthesize** transforms those observations into a structured configuration using Pydantic structured outputs. This was a deliberate decision. An earlier version parsed JSON manually with `json.loads` and an `except` that returned `{"raw": raw}` on failure, silently propagating a broken configuration downstream. With structured outputs, the contract is airtight: if the model does not produce a valid `AgentConfig`, it fails explicitly. Field validators add semantic validation. A `system_prompt` of 10 characters is structurally valid but useless.

**Generate** takes the configuration, maps the tools requested by the stem to the tools that actually exist in the system, validates the result with `ExecutableAgentConfig`, and persists everything to disk along with a trace file containing the full observations, timing, git hash, and timestamp. Persistence eliminates regeneration cost. The stem phase takes 3 to 4 minutes with GPT-5 and only runs once.

The agent executes tasks using the ReAct pattern: Thought, Action, Observation in a loop until a final answer is reached. Before each task, an LLM classifier decides whether the task belongs to the agent's domain. The `accepts` and `rejects` lists are natural language generated by the stem. Only an LLM can reason about whether a task fits.

The tool pool is currently fixed at two tools, `web_search` and `run_code`, but the architecture supports expansion without structural changes. Adding a new tool means adding a function to `AGENT_TOOLS` in `agent/tools.py`. Because `stem/schemas.py` derives `ToolName` directly from that registry, the stem discovers the new tool automatically on the next transformation. A QA agent could request `run_code` and `read_file`. A Security agent could request `fetch_url` and `run_code`. The stem reasons about what it needs. The system provides what exists.

## Experiments and what did not go as expected

**The model cascade.** Observation with GPT-5 took ~90 seconds versus ~9 seconds with GPT-4o, but produced significantly richer observations. My hypothesis was that a cascade, GPT-4o for fast exploration, GPT-5 for deepening, would reduce time while maintaining quality. I implemented the cascade, measured it, and the result was the opposite: 157 seconds, more than GPT-5 alone. The problem was that GPT-5 received more input tokens (the GPT-4o exploration on top of the domain) which increased processing time. I discarded the cascade and used GPT-5 directly. The stem phase runs once per domain, so the time cost is amortizable.

**Tool matching.** The stem generated tools in natural language: "Google Scholar", "ArXivSearch", "ExperimentTrackerDesigner". I implemented three different versions to map those names to available tools: first keyword heuristics with substring matching, then an LLM for semantic matching, and finally exact matching. The LLM matching added latency and an extra failure point with no real benefit, because the correct solution was to inform the stem of the available pool in the synthesis prompt. Once the stem knew the exact names, trivial matching was sufficient. Three iterations to arrive at the simplest solution.

**The domain classifier.** The first version of the classifier rejected legitimate tasks. The problem was that the rejects generated by the stem included "Executing tools or network calls beyond web_search". This was designed to prevent access to external systems, but semantically ambiguous. The classifier read "Executing tools" and rejected code execution tasks. The fix was to add an explicit rule to the classifier prompt: available tools are always permitted, regardless of the rejects list. This revealed a deeper limitation: the `accepts` and `rejects` generated by the stem describe the agent's capabilities, not criteria for classifying incoming tasks. They are conceptually distinct.

## Results and analysis

The evaluation compares the auto-generated agent against a minimal baseline, "You are a helpful assistant" with the same tools. Averaged over 3 runs:

| Metric | Baseline | Stem Agent |
|---|---|---|
| Avg quality score simple (1-5) | 4.60 | 4.33 |
| Avg quality score complex (1-5) | 4.56 | 4.56 |
| Rejection accuracy | 0% | 100% |
| False acceptance rate | 100% | 0% |

To ensure a fair comparison (an ablation study), the baseline is not a bare LLM. It uses the same ReAct engine and the same tools (run_code, web_search), but with a generic prompt and no domain classifier. This isolates the exact value of the stem pipeline.

The stem agent does not improve response quality as measured by an LLM judge (GPT-4o). Both agents produce responses of similar quality, and on simple tasks the baseline scores slightly higher. The explanation is that the system prompt generated by the stem is optimized for rigorous research methodology, which introduces overhead on tasks that need a direct answer. On complex tasks involving experimental design and trade-off analysis, scores are identical.

What the stem demonstrates unambiguously is specialization: it rejects 100% of out-of-domain tasks while the baseline accepts everything. An agent that answers cooking recipes and vulnerability analyses with equal confidence is not specialized. It is simulating specialization. Specialization has value in production not because it improves each individual response, but because it makes the system predictable.

---

## What failed

The LLM judge does not discriminate research quality well enough. It evaluates whether the response is correct and complete, but not whether it follows rigorous methodology, such as references to confidence intervals, experimental design, threats to validity. A human evaluator would see differences the automated judge misses. This limits the benchmark's ability to demonstrate the value of the specialized system prompt.

The base model already has enough technical research knowledge to respond well without a specialized system prompt. GPT-4o in "helpful assistant" mode produces responses that an LLM judge cannot distinguish from those of the specialized agent. The stem adds structure and methodological rigor, but that rigor is not captured by current metrics.

---

## With more time

I would implement a more discriminative judge that evaluates methodological rigor rather than just correctness: whether the response defines concrete success metrics, mentions baselines, quantifies uncertainty, identifies threats to validity, and proposes reproducible steps. With that judge, the stem agent would likely win on complex tasks because its system prompt explicitly instructs it to do all of those things.

I would explore conditional routing in the observe phase: use GPT-4o by default and escalate to GPT-5 only if the output does not pass a quality threshold. The cascade I tested failed because it added unnecessary context. Conditional routing would avoid that problem.

I would add adaptive prompts. The stem could generate two modes: a concise mode for direct execution tasks and a rigorous mode for analysis tasks. The agent would choose the mode based on the nature of the task.

The project demonstrates that an agent can build itself from environmental signals. The pattern is clear: the programmer defines the domain, but does not write the agent. The stem observes, reasons, and decides how to specialize. That separation makes it a natural building block in multi-agent systems where an orchestrator can instantiate specialized agents on demand, by simply changing the domain input.
