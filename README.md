# Stem Agent

A minimal agent that receives a class of problems, observes how experts work in that domain, and builds itself into a specialized agent without any hardcoded domain knowledge.

The specialized agent is not written by the programmer. It is generated by the stem.

---

## How it works

```
Input: "Technical Research"
        ↓
[STEM PHASE]
  observe    → researches how experts work in the domain
  synthesize → extracts patterns and decides agent architecture
  generate   → produces a validated, executable agent configuration
        ↓
[EXECUTION PHASE]
  ReAct loop → Thought → Action → Observation → repeat
  Domain classifier → accepts in-domain tasks, rejects out-of-domain
```

The stem phase runs once per domain and persists the result to disk. Subsequent runs load the configuration instantly.

## Setup

**Requirements:** Python 3.11+

```bash
git clone https://github.com/josegijon/stem-agent.git
cd stem-agent

python -m venv .venv

# Windows
.venv\Scripts\activate

# macOS / Linux
source .venv/bin/activate

pip install -r requirements.txt
```

Copy `.env.example` to `.env` and fill in your values:

```bash
cp .env.example .env
```

```dotenv
OPENAI_API_KEY=your_key_here
OPENAI_MODEL_DEEP=gpt-5        # used for observe and synthesize
OPENAI_MODEL_FAST=gpt-4o       # used for ReAct loop and tool calls
STEM_DOMAIN=Technical Research
LOG_LEVEL=INFO
REACT_MAX_STEPS=12
BENCHMARK_RUNS=3
DEBUG=false
```

## Usage

### Run the stem agent

```bash
python main.py
```

The first run triggers the stem transformation (~3-4 minutes with GPT-5). Subsequent runs load the generated config from disk and start instantly.

```
============================================================
        STEM AGENT
============================================================

[STEM] Domain: Technical Research
[STEM] Found existing config. Skipping transformation.

[AGENT] Specialized agent ready: Decision-driven Technical Research Orchestrator

Task: What are the trade-offs between vector databases and traditional databases for semantic search?
```

### Force regeneration

Delete the generated config to trigger a new stem transformation:

```bash
# Windows
del config\generated\technical_research.json

# macOS / Linux
rm config/generated/technical_research.json
```

### Run the evaluation benchmark

```bash
python -m eval.benchmark
```

Runs the stem agent and baseline against the same task set and prints a comparison table.

## Evaluation results

Averaged over 3 runs.

| Metric | Baseline | Stem Agent |
|---|---|---|
| Avg steps (in-domain) | 1.6 | 2.7 |
| Avg quality score (1-5) | 4.60 | 4.33 |
| Rejection accuracy | 0% | 100% |
| False acceptance rate | 100% | 0% |
| Avg quality complex tasks (1-5) | 4.56 | 4.56 |

**What this means:**

The stem agent maintains comparable response quality on in-domain tasks while achieving perfect domain specialization. The baseline accepts and answers everything (unit tests, recipes, legal advice) with equal confidence. The stem agent rejects 100% of out-of-domain tasks.

The quality gap on simple tasks is explained by the generated system prompt being optimized for rigorous research methodology, which introduces overhead on tasks that need a direct answer. On complex research tasks (experimental design, research critique, trade-off analysis), both agents score identically.

## Design decisions worth noting

**Why ReAct and not Plan-and-Execute?**
ReAct decides the next step after observing the result of the previous one. If a tool fails or returns unexpected output, the agent adapts. Plan-and-Execute commits to a fixed plan upfront and cannot recover.

**Why GPT-5 for the stem phase and GPT-4o for execution?**
The stem phase runs once. Quality of observations directly determines quality of the generated agent. GPT-5 produces richer, more structured observations. GPT-4o handles the ReAct loop which runs on every task — speed and cost matter there.

**Why exact tool matching instead of LLM-based matching?**
The stem is informed of available tools in the synthesize prompt and requests them by exact name. A previous implementation used LLM-based semantic matching, but it added latency and a failure point for no benefit once the stem knew the tool names.

**Why persist the generated config to disk?**
The stem phase costs ~3-4 minutes and ~$0.10 per run with GPT-5. Persisting eliminates redundant runs. The trace file saves the full provenance (observations, timing, git hash) so any run is reproducible without calling the API again.

## Trace file

Every stem run produces a trace file at `config/generated/<domain>_trace.json` with:

- Full observations from the observe phase
- Model and timing for each phase
- Tool matching: requested, matched, discarded
- Git commit hash and UTC timestamp
- Final agent configuration

## Known limitations

- `web_search` uses the LLM as a search proxy. It responds from training knowledge, not live web retrieval. Results may be outdated for recent events.
- The generated system prompt is optimized for rigorous research methodology. On simple tasks, a shorter prompt produces more direct answers.
- The stem phase is non-deterministic, two runs for the same domain may produce different configurations. GPT-5 does not support `temperature=0`.
- Domain classification depends on the quality of `accepts` and `rejects` generated by the stem. Ambiguous rejects can cause false negatives in the classifier.

## What I would do with more time

- **Routing conditional model cascade**: use GPT-4o for observe and only escalate to GPT-5 if the output fails a quality threshold. Reduces cost and latency for simple domains.
- **Structured observe output**: instead of free text, observe could return a typed schema. Reduces the interpretation burden on synthesize and produces more consistent configs.
- **Adaptive system prompts**: the stem could generate different prompt modes, a concise mode for direct tasks and a rigorous mode for research tasks, and let the ReAct loop switch between them.
- **Real web search**: replace the LLM search proxy with a real search API (Serper, Brave) to get current information.
- **Human evaluation**: the LLM judge measures correctness but not research quality. A human evaluator would surface differences the automated judge misses.
